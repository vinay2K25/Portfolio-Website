01

Backend for Spotify

C Data Structures Algorithms
What does it do?

This project is a command-line backend simulation of a music streaming player. It allows users to create playlists, manage songs, maintain a playback queue, and control playback using commands such as play, pause, next, and previous. The system persists playlists to disk so they can be reloaded across sessions.

Internally, the application uses linked data structures to represent playlists, songs, and queues, allowing efficient insertion, deletion, and traversal operations.

Why I built it?

This project was built to apply core data structure concepts in a practical setting. By implementing playlists, queues, and playback controls using linked lists, I explored dynamic memory management and traversal beyond theory. It also served as a way to simulate the backend logic of a music player, including command-driven interaction and file-based persistence across sessions.

How would I do it differently now?

If I were rebuilding the project today, I would focus on improving the architecture and separation of concerns. For example, I would introduce clearer abstractions for playlist management, playback state, and storage. I would also implement better memory safety checks, stronger input validation, and possibly convert the storage format to something structured such as JSON rather than a custom text format.

Trade-offs

The design favors simplicity over scalability. Linked lists make insertions and deletions efficient, but searching remains linear and less optimal for large datasets. The use of a command-line interface and plain-text storage keeps the system easy to implement, but limits user experience and robustness compared to structured storage and more advanced architectures.

02

TwixT Game

C Algorithms Data Structures
What does it do?

This project is a command-line implementation of the strategy board game TwixT, written in C. The program allows two players to take turns placing pegs on a 24×24 board and automatically creates valid links between pegs using the game's knight-move rule. The system enforces game constraints such as preventing illegal placements and detecting link intersections. A depth-first search based path-checking mechanism determines whether a player has successfully connected their sides of the board to win the game.

Why I built it?

I built this project to explore how graph traversal and geometric constraints can be applied to real game mechanics. TwixT is an interesting problem because it combines board representation, intersection detection, and path-finding. Implementing it from scratch allowed me to practice structuring a multi-file C project, managing game state, and designing algorithms that enforce the rules of a non-trivial strategy game.

How would I do it differently now?

If I were rebuilding the project today, I would improve the architecture and modularity of the system. For example, I would separate the game logic, board representation, and input/output layers more clearly. I would also introduce better abstractions for links and graph traversal, possibly representing the board as an explicit graph structure rather than relying on coordinate arrays. Additionally, I would add a graphical interface or web-based visualization to make the game easier to interact with.

Trade-offs

While building the project, I prioritized simplicity and clarity of the game logic over extensibility. As a result, some structures such as global link storage and fixed-size arrays make the implementation straightforward but less flexible. The command-line interface also simplified development but limits usability compared to a graphical interface. These trade-offs allowed the core mechanics — link creation, intersection checking, and win detection — to be implemented efficiently within a manageable codebase.

03

Task Planner

Flask SQLite Python Web
What does it do?

This project is a lightweight web-based task manager built using Flask and SQLite. It allows users to add, complete, and delete tasks through a simple interface. Tasks are stored persistently in a SQLite database, enabling users to maintain their task list across sessions. The application demonstrates basic CRUD operations through REST-style routes and dynamically renders tasks using a template-based frontend.

Why I built it?

I built this project to understand the fundamentals of backend web development. It helped me learn how web servers handle routes, how databases integrate with applications, and how data flows between the frontend and backend. The project also served as a practical introduction to Flask and database-driven web applications.

How would I do it differently now?

If I were rebuilding this project today, I would structure it using a clearer separation between the backend logic, database layer, and frontend templates. I would also add features such as task editing, categories, authentication, and asynchronous updates using JavaScript or an API-driven approach. Additionally, I would likely use an ORM like SQLAlchemy for safer and more scalable database interactions.

Trade-offs

During development, I focused on keeping the project simple and easy to understand. As a result, the application directly interacts with the database in each route instead of using an abstraction layer. While this approach reduces complexity for a small project, it can make scaling and maintenance more difficult as the application grows. The minimal interface also prioritizes functionality over advanced user experience features.
